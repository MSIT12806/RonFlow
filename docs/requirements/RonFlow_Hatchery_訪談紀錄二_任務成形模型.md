# RonFlow Hatchery 訪談紀錄二：任務成形模型

## 一、訪談目的

本次訪談為 Hatchery 第二次需求訪談，主題是「任務成形模型」。

本次訪談的目的，是定義 RonFlow 要如何描述一個任務尚未成形、正在拆解、已經拆解完成、已經 Ready，以及進入既有 Flow 後的狀態關係。

本次訪談不進入畫面設計、資料表設計或 API 設計，而是先釐清以下模型問題：

```text
1. RonFlow 裡的核心名詞如何定義？
2. 任務成熟度狀態有哪些？
3. 父任務與葉任務的規則是否不同？
4. 哪些任務可以進入既有 Flow？
5. Ready checklist 應包含哪些條件？
6. 不 Ready 的任務要如何處理？
7. 狀態轉換是否允許倒退？
```

---

## 二、訪談結論摘要

本次訪談得到的核心結論如下：

```text
1. 專案與任務要分開。
2. 專案是工作區域，不是任務。
3. Hatchery 管理的是專案底下的任務樹。
4. 所有節點在介面上都先稱為「任務」。
5. 沒有子任務的任務稱為「葉任務」。
6. 新建立的任務預設視為葉任務。
7. 葉任務需要填寫 Ready checklist。
8. 使用者若建立子任務，原任務就轉為父任務。
9. 父任務不用填 Ready checklist。
10. 父任務走拆解狀態；葉任務走 Ready / Flow 狀態。
11. 只有葉任務可以進入既有 Flow。
12. Ready 任務進入 Flow 後，仍是同一個任務，不建立副本。
13. Hatchery 階段只看成熟度；進入 Flow 後只看 Flow 狀態。
14. 不處理不 Ready 原因；只判斷是否符合 Ready 條件。
15. MVP 不做任務分類、標籤、AI 判讀、視覺化、報表、跨專案、多人協作或自動估時。
```

---

## 三、Domain Glossary

### 1. Project：專案

專案是 RonFlow 中的工作區域，用來承載某一組相關任務。

```text
專案不是任務。
專案不是超大型任務。
專案不進入 Flow。
專案是任務樹所在的容器與脈絡。
```

---

### 2. Task：任務

RonFlow 介面上統一使用「任務」這個名稱。

不論它目前是大型任務、父任務、子任務、葉任務，介面上都先稱為任務，避免過早引入過多名詞，例如目標、工作包、Epic、階段等。

原因是：

```text
如果系統要區分「目標」與「任務」，就需要定義什麼時候目標會變成任務。
這會引入額外界線問題，反而增加模型複雜度。
```

因此 MVP 中先統一稱為「任務」。

---

### 3. Parent Task：父任務

父任務是擁有子任務的任務節點。

一個任務只要被建立子任務，它就從葉任務轉為父任務。

父任務的重點不是直接執行，而是管理拆解結構與子任務完成情況。

父任務：

```text
1. 可以被標記為拆解完成。
2. 不需要填寫 Ready checklist。
3. 不直接進入既有 Flow。
4. 所有子任務完成後，父任務自動完成。
5. 父任務完成狀態需要保留，以利後續統計，例如完成時間、複雜度等。
```

---

### 4. Leaf Task：葉任務

中文介面名稱：葉任務。

葉任務是沒有子任務的任務。

新建立的任務預設都視為葉任務。

葉任務的特徵：

```text
1. 沒有子任務。
2. 需要填寫 Ready checklist。
3. 若符合 Ready 條件，可以進入既有 Flow。
4. 若使用者新增子任務，該任務會轉為父任務。
```

---

### 5. Task Tree：任務樹

任務樹是專案底下的任務拆解結構。

任務可以遞迴建立子任務，不限制層數。

```text
Project
  └── Task Tree
        ├── Task
        │    ├── Task
        │    └── Leaf Task
        └── Leaf Task
```

底層資料模型應支援樹狀結構。

大綱結構、樹狀結構、Mind Map 在資料結構上都可以基於同一套任務樹，只是呈現方式不同。

MVP 先不做 Mind Map 或其他視覺化，只需要支援基本樹狀／大綱式拆解。

---

### 6. Ready Task：Ready 任務

Ready 任務是符合 Ready checklist 的葉任務。

Ready 任務代表：

```text
這個任務已經足夠明確，可以進入既有 Flow 被執行。
```

Ready 僅適用於葉任務，不適用於父任務。

---

### 7. Flow Task：進入 Flow 的任務

Ready 任務可以被送入 RonFlow 既有 Flow。

進入 Flow 後，不建立副本，也不從任務樹移除，而是同一個任務開始具有 Flow 狀態。

使用者視角上，可以理解為：

```text
同一個任務，原本在 Hatchery 中成形；
符合 Ready 條件後，進入 Flow 中執行。
```

---

### 8. Split Complete：拆解完成

拆解完成是父任務的狀態。

它表示：

```text
這個父任務已經被拆到可接受程度，不需要繼續拆解。
```

拆解完成不等於任務完成。

例如：

```text
「設計 Hatchery 任務成形模型」可能已經拆解完成，
但它底下的子任務尚未全部完成。
```

---

### 9. Done：完成

完成是任務執行完成的狀態。

父任務與葉任務都需要保留完成狀態。

原因是未來可能需要統計：

```text
1. 父任務完成時間。
2. 任務樹完成時間。
3. 大型任務複雜度。
4. 任務拆解深度。
5. 任務實際完成歷程。
```

---

## 四、任務成熟度與 Flow 狀態的關係

### 1. Hatchery 階段與 Flow 階段分離

本次訪談確認採用以下方向：

```text
Hatchery 階段只看成熟度。
進入 Flow 後只看 Flow 狀態。
```

也就是說，任務在 Hatchery 中主要關心：

```text
是否需要拆解？
是否已拆解完成？
是否符合 Ready 條件？
是否可以送進 Flow？
```

任務進入 Flow 後，主要關心既有 Flow 狀態，例如：

```text
Todo
Active
Review
Done
```

---

### 2. 不在畫面上同時暴露兩套狀態

雖然系統內部可能需要記錄 Hatchery 狀態與 Flow 狀態，但使用者不應被迫同時理解兩套狀態。

使用者視角應是：

```text
在 Hatchery 中：看任務是否還需要拆解、是否 Ready。
進入 Flow 後：看任務在 Todo / Active / Review / Done 的執行進度。
```

---

## 五、父任務狀態模型

父任務是擁有子任務的任務。

父任務不填寫 Ready checklist，也不直接進入 Flow。

父任務建議狀態如下：

```text
拆解中
  ↓
拆解完成
  ↓
所有子任務完成
  ↓
已完成
```

### 1. 拆解中

當任務擁有子任務，但尚未標記拆解完成時，視為拆解中。

此狀態表示：

```text
這個任務仍在被規劃與拆解。
```

---

### 2. 拆解完成

使用者可手動標記父任務為拆解完成。

此狀態表示：

```text
這個父任務暫時不需要再繼續拆解。
```

但它不代表所有子任務已完成。

---

### 3. 已完成

父任務採用自動完成規則：

```text
所有子任務完成後，父任務自動完成。
```

此規則可支援後續統計。

---

## 六、葉任務狀態模型

葉任務是沒有子任務的任務。

新建立的任務預設視為葉任務。

葉任務建議狀態如下：

```text
未 Ready
  ↓ 填寫並確認 Ready checklist
Ready
  ↓ 送進 Flow
進入 Flow
  ↓ Flow 完成
已完成
```

---

### 1. 未 Ready

新建立的葉任務預設為未 Ready。

使用者需要先規劃它的產出物、完成條件與預估時間。

---

### 2. Ready

當葉任務符合 Ready checklist 後，可以標記為 Ready。

Ready 表示：

```text
這個任務已經足夠明確，可以進入 Flow。
```

---

### 3. 進入 Flow

Ready 任務可以被送入既有 Flow。

進入 Flow 後，任務改由既有 Flow 狀態管理。

---

### 4. 已完成

當任務在 Flow 中完成後，該葉任務也視為已完成。

---

## 七、Ready checklist

MVP 的 Ready checklist 暫定為三項：

```text
1. 是否有明確產出物？
2. 是否有完成條件？
3. 是否可以明確預估預計完成時間？
```

---

### 1. 明確產出物

使用者需要先描述此任務完成後會留下什麼產出。

例如：

```text
一份 Hatchery 任務成形模型文件。
一份 Ready checklist 定義表。
一個可運作的任務樹新增功能。
一組前端表單欄位。
```

---

### 2. 完成條件

使用者需要定義怎樣算完成。

例如：

```text
文件中包含成熟度狀態、Ready checklist、狀態轉換規則。
功能可新增子任務、顯示父子層級、保留任務排序。
表單可填寫產出物、完成條件、預估時間，並可確認 Ready。
```

---

### 3. 預計完成時間

使用者需要填寫可明確預估的預計完成時間。

時間格式採用：

```text
數字 + 單位
```

例如：

```text
2 小時
0.5 天
1 天
3 天
```

目前 RonFlow 已有 reminder 機制，後續可搭配 reminder 製作預計完成時間或到期時間。

---

### 4. Ready checklist 的互動方式

Ready checklist 採用以下流程：

```text
1. 使用者先填寫規劃內容。
2. 使用者確認填寫內容足夠明確。
3. 使用者再將該項 checklist 勾選起來。
```

也就是說，不是單純 checkbox，而是：

```text
先填文字或時間欄位，再由使用者確認 checked。
```

此設計的原因是：

```text
如果只有 checkbox，無法留下任務成形資訊。
如果只有文字欄位，缺少明確確認動作。
```

因此 MVP 採用：

```text
填寫欄位 + 使用者確認勾選
```

---

## 八、是否處理不 Ready 原因

本次訪談確認：MVP 不處理不 Ready 原因。

原因是第一版只需要判斷：

```text
哪些任務符合 Ready 條件。
```

其他任務都視為不 Ready。

不 Ready 任務的處理方式是：

```text
1. 保留在任務樹中。
2. 使用者可以補齊 Ready checklist。
3. 使用者也可以繼續拆解子任務。
```

MVP 不需要額外建立不可執行原因清單，例如：

```text
任務太大
缺少產出物
缺少完成條件
無法預估時間
需要調查
需要決策
等待依賴
```

這些原因可以留到後續版本再評估。

---

## 九、任務分類與標籤

MVP 不做任務分類或標籤功能。

也就是說，第一版不要求使用者標記：

```text
調查
決策
設計
實作
驗收
文件
溝通
```

原因是：

```text
1. 任務分類會增加使用者輸入成本。
2. 第一版核心目標是任務拆解與 Ready 檢核。
3. 調查、決策、設計等性質可以先透過任務名稱與完成條件表達。
```

例如：

```text
調查有哪些任務成熟度模型可以參考
```

不需要額外標籤為「調查」。

只要它有明確產出物、完成條件與預估時間，它就是可以進入 Flow 的 Ready 任務。

---

## 十、任務進入 Flow 的規則

### 1. 只有葉任務可以進入 Flow

規則：

```text
有子任務的父任務：不能進 Flow。
沒有子任務的葉任務：通過 Ready checklist 後，可以進 Flow。
```

理由：

```text
父任務通常代表工作包、方向或拆解節點。
若父任務進入 Flow，容易造成完成定義不清，也可能長期卡在 Active。
```

因此，Flow 只處理真正可執行的葉任務。

---

### 2. 父任務本身若有產出，必須拆成子任務

如果一個父任務本身也有工作內容或產出，應另拆子任務表示。

例如：

```text
設計 Hatchery 任務成形模型
  ├── 定義成熟度狀態
  ├── 定義 Ready checklist
  ├── 定義不可執行原因
  └── 撰寫任務成形模型文件
```

也就是說：

```text
父任務不能自己進 Flow。
若父任務本身有可交付產出，應拆出一個葉任務承擔該產出。
```

---

### 3. Ready 任務進入 Flow 後仍是同一個任務

Ready 任務送進 Flow 後，不複製、不移除、不另建一筆。

採用規則：

```text
同一個任務，只是多了一個 Flow 狀態。
```

原因：

```text
1. 避免任務樹與 Flow 出現資料重複。
2. 保留任務從 Hatchery 到 Flow 的完整歷程。
3. 方便父任務統計子任務完成情況。
```

---

## 十一、狀態轉換規則

### 1. 父任務狀態轉換

```text
新任務
  ↓ 使用者建立子任務
父任務 / 拆解中
  ↓ 使用者標記拆解完成
拆解完成
  ↓ 所有子任務完成
已完成
```

---

### 2. 葉任務狀態轉換

```text
新任務 / 葉任務 / 未 Ready
  ↓ 填寫 Ready checklist 並確認
Ready
  ↓ 送進 Flow
進入 Flow
  ↓ Flow 完成
已完成
```

---

### 3. 新建立任務的預設規則

本次訪談確認以下畫面與行為想像：

```text
新建立出來的任務都視為葉任務。
因此新任務預設需要輸入 Ready checklist。
除非使用者按下「建立子任務」，此時原本的任務就會變成父任務，不再輸入 Ready checklist。
```

此規則可以讓系統簡化判斷：

```text
沒有子任務 = 葉任務 = 檢查 Ready checklist
有子任務 = 父任務 = 檢查拆解狀態與子任務完成情況
```

---

### 4. 狀態是否允許倒退

MVP 暫時允許狀態倒退，但先不做完整紀錄。

例如：

```text
Ready → 未 Ready
拆解完成 → 拆解中
進入 Flow → 退回任務樹
```

此方向的原因是：

```text
需求與任務拆解可能會變動。
第一版不應把狀態做得過於剛性。
```

後續版本可再補狀態異動紀錄。

---

### 5. Ready 任務新增子任務時的規則

如果一個已 Ready 的葉任務後來新增子任務，表示使用者發現它其實仍然太大或需要拆解。

採用規則：

```text
允許新增子任務，但要使用者確認狀態改回拆解中。
```

也就是：

```text
Ready Leaf Task
  ↓ 使用者新增子任務並確認
Parent Task / 拆解中
```

此規則避免出現矛盾狀態：

```text
一個任務同時是 Ready 葉任務，又是擁有子任務的父任務。
```

---

## 十二、MVP 模型總結

MVP 的任務成形模型可以總結如下：

```text
Project
  └── Task Tree
        ├── Parent Task
        │     ├── Leaf Task → Ready checklist → Flow
        │     └── Leaf Task → Ready checklist → Flow
        └── Leaf Task → Ready checklist → Flow
```

---

## 十三、MVP 必要規則

MVP 至少需要支援以下規則：

```text
1. 專案是工作區域，不是任務。
2. 專案底下可以建立任務樹。
3. 任務可遞迴建立子任務，不限制層數。
4. 新任務預設為葉任務。
5. 葉任務需要填 Ready checklist。
6. 葉任務符合 Ready checklist 後可以進 Flow。
7. 任務一旦新增子任務，就成為父任務。
8. 父任務不填 Ready checklist。
9. 父任務不能進 Flow。
10. 父任務可標記拆解完成。
11. 所有子任務完成後，父任務自動完成。
12. Ready 任務進入 Flow 後仍是同一個任務。
13. 不 Ready 任務保留在任務樹中。
14. MVP 不處理不 Ready 原因。
15. MVP 不做任務類型或標籤。
```

---

## 十四、MVP 非目標

第一版不做以下功能：

```text
AI 自動拆解
AI Ready 判讀
不 Ready 原因管理
任務類型分類
標籤功能
Mind Map 視覺化
報表統計
跨專案依賴
多人協作
自動估時
複雜甘特圖
任務難度自動判斷
完整狀態異動紀錄
```

---

## 十五、後續待釐清問題

雖然本次訪談已經完成任務成形模型的主要輪廓，但後續仍有一些問題需要在第三、第四次訪談中釐清：

```text
1. 任務樹在畫面上如何呈現？
2. Ready checklist 表單在任務細節中如何操作？
3. Ready 任務如何送進 Flow？
4. Flow 中的任務如何回到任務樹查看上下文？
5. 父任務完成進度是否需要在 MVP 中顯示？
6. 拆解完成按鈕與完成狀態在畫面上如何區分？
7. 預計完成時間與 reminder / due date 的關係如何設計？
8. 已進入 Flow 的任務若被退回 Hatchery，既有 Flow 狀態如何處理？
9. 父任務自動完成後，是否允許再次新增子任務？
10. 任務樹排序、拖曳、縮排、移動父節點是否進 MVP？
```

## 十六、可作為驗收案例的主案例

後續驗收案例使用：

```text
RonFlow 需求說明書：抽象任務分解與任務成形管理
```

可驗證流程如下：

```text
1. 在專案中建立「RonFlow Hatchery」任務。
2. 新任務預設為葉任務，顯示 Ready checklist。
3. 使用者建立子任務，使其轉為父任務。
4. 父任務底下拆出「定義任務成形模型」、「設計使用者流程」、「規劃 MVP 切片」等子任務。
5. 子任務可繼續拆解。
6. 某個葉任務填寫產出物、完成條件、預估時間。
7. 使用者確認 checklist，該葉任務變成 Ready。
8. Ready 葉任務送入既有 Flow。
9. Flow 中完成該任務後，任務樹同步反映完成狀態。
10. 所有子任務完成後，父任務自動完成。
```

---

## 十七、本次訪談結論

本次訪談完成了 Hatchery 的任務成形模型初步定義。

最重要的模型決策是：

```text
Hatchery 不取代 Flow。
Hatchery 負責讓任務成形。
Flow 負責執行已 Ready 的葉任務。
```

也可以表述為：

```text
任務樹管理拆解與成形；
Ready checklist 判斷葉任務是否可執行；
Flow 管理可執行任務的實際推進。
```

MVP 應聚焦於：

```text
遞迴任務拆解
父任務與葉任務區分
Ready checklist
Ready 葉任務進 Flow
父任務完成統計基礎
```

不應在第一版處理：

```text
AI、視覺化、報表、多人協作、跨專案、複雜分類、完整狀態歷程。
```
