# RonFlow 需求說明書：抽象任務分解與任務成形管理

## 一、背景

RonFlow 目前是一套用來管理流程與任務執行的系統。

在目前的使用情境中，RonFlow 主要處理的是已經被規劃好、可以直接落地執行的任務。也就是說，任務進入 RonFlow 時，通常已經具備一定程度的明確性：使用者知道要做什麼、知道任務完成後會產生什麼成果，也能判斷任務是否完成。

然而，在實際工作中，並不是所有任務一開始都具備這種清晰度。

有些任務本身很大、很抽象，也包含大量未知，例如：

```text
設計一個分散式匯率查詢系統
```

這類任務無法直接進入執行流程。它不是一個可以馬上實作的任務，而是一個尚未成形的工作。使用者必須先釐清目標、理解限制、拆解系統、做出架構決策，最後才能產生可以實作的任務。

因此，RonFlow 需要從「管理已經可以執行的任務」進一步擴展為：

```text
幫助使用者把模糊、抽象、大型的工作，逐步分解成可以被執行、被驗收、被追蹤的任務。
```

------

## 二、目前困難

### 1. 抽象任務無法直接執行

目前 RonFlow 面對的任務多半是假設已經被整理過的任務。

但實際上，使用者常常遇到的是這種任務：

```text
建立一套會員系統
設計一個分散式匯率查詢系統
改善工程團隊產出衡量方式
從零開始經營 YouTube 頻道
規劃一個新的產品功能
```

這些任務本身不是「可執行任務」，而是「需要被拆解的工作」。

如果直接把它們放進一般任務流程，會產生幾個問題：

- 任務過大，不知道如何開始。
- 完成條件不明確，不知道做到什麼程度才算完成。
- 任務中包含太多決策，執行過程會不斷卡住。
- 任務中包含大量未知，導致無法估時、排程或驗收。
- 任務看似在流程中移動，但實際上沒有真正產生進展。

------

### 2. 任務管理工具通常只管理「狀態」，不管理「成熟度」

一般任務流程通常關注的是：

```text
Todo → Doing → Done
```

也就是任務目前做到哪裡。

但抽象任務真正需要處理的是另一個問題：

```text
模糊 → 已定義 → 已拆解 → 可執行 → 已完成
```

這是任務的成熟度問題。

一個任務可能還在 Backlog，但它已經非常清楚，可以隨時執行。

另一個任務也可能在 Backlog，但它其實還只是一個模糊想法，根本不能被執行。

如果 RonFlow 只管理任務狀態，而不管理任務成熟度，就無法有效處理這類抽象工作。

------

### 3. 使用者需要管理「不確定性」，而不只是管理待辦事項

抽象任務之所以難以執行，通常不是因為使用者沒有動手，而是因為任務中存在大量未知。

未知可能包含：

```text
需求未知
範圍未知
技術未知
依賴未知
風險未知
成本未知
優先順序未知
完成條件未知
驗收標準未知
```

例如「設計一個分散式匯率查詢系統」可能包含以下未知：

```text
資料來源從哪裡來？
是否需要即時匯率？
可以接受多大的資料延遲？
外部 API 掛掉時要怎麼辦？
是否需要快取？
是否需要支援歷史匯率？
系統需要多高可用性？
服務邊界要怎麼切？
```

這些問題沒有被釐清前，任務很難直接進入實作。

因此，RonFlow 應該幫助使用者識別未知、管理未知，並透過調查、決策、設計等工作逐步降低不確定性。

------

## 三、問題本質

目前的核心問題不是 RonFlow 缺少某個頁面或某個按鈕，而是 RonFlow 的任務模型需要擴展。

原本 RonFlow 管理的是：

```text
已經成形的任務如何被完成
```

但現在需要補上的能力是：

```text
尚未成形的工作如何變成可以執行的任務
```

換句話說，RonFlow 不應該只是一個任務執行工具，也應該是一個任務成形工具。

RonFlow 需要同時回答三個問題：

```text
這件事被拆成什麼結構？
這件事是否已經夠清楚？
這件事現在做到哪裡？
```

其中：

```text
結構問題：需要任務分解能力
清晰度問題：需要任務成熟度管理
進度問題：需要既有工作流程管理
```

------

## 四、RonFlow 的目標

RonFlow 的目標是幫助使用者將大型、抽象、模糊的工作，逐步分解、釐清、決策，最後轉化成可執行任務。

可以用一句話描述：

```text
RonFlow 幫助使用者把模糊目標轉化成可執行任務，並追蹤任務從成形到完成的整個過程。
```

更具體地說，RonFlow 應該幫助使用者完成以下事情：

```text
1. 捕捉尚未成形的大型任務或模糊目標。
2. 判斷一個任務目前是否足夠清楚。
3. 找出任務中的未知、依賴與重大決策。
4. 將大型任務拆解成較小的工作項目。
5. 區分調查、決策、設計與實作等不同性質的工作。
6. 判斷哪些工作已經可以進入執行流程。
7. 將可執行任務流入既有的 RonFlow 任務流程。
```

RonFlow 的前半段負責讓任務「變清楚」，後半段負責讓任務「被完成」。

------

## 五、RonFlow 對任務的核心觀點

RonFlow 應該將任務視為一個會逐漸成熟的工作項目，而不是一開始就假設所有任務都已經可以執行。

一個工作項目可能經歷以下變化：

```text
模糊想法
  ↓
初步定義
  ↓
拆解成子項目
  ↓
釐清未知與依賴
  ↓
做出必要決策
  ↓
形成可執行任務
  ↓
進入執行流程
  ↓
完成與驗收
```

因此，RonFlow 不只需要記錄任務目前的執行狀態，也需要記錄任務目前的成熟程度。

------

## 六、任務夠小的判斷依據

RonFlow 應該內建一套判斷任務是否足夠小、是否可以執行的標準。

一個任務若要進入執行流程，至少應該符合以下條件：

```text
1. 有明確動詞。
2. 有具體產出物。
3. 有完成條件。
4. 可以在合理時間內完成。
5. 執行時不需要再做重大決策。
6. 依賴關係清楚。
7. 失敗成本可控。
```

也可以簡化成五個核心問題：

```text
1. 我知道要做什麼嗎？
2. 我知道第一步是什麼嗎？
3. 我知道完成後會留下什麼嗎？
4. 我知道怎樣算完成嗎？
5. 我能在合理時間內完成它嗎？
```

如果這些問題無法回答，代表這個工作項目還不應該直接進入執行流程，而應該繼續被拆解、釐清或轉化成調查與決策任務。

------

## 七、工作項目的不同性質

RonFlow 需要承認一件事：

```text
不是所有任務都是實作任務。
```

在大型抽象任務中，常常會產生不同性質的工作。

### 1. 調查任務

用來降低未知。

例如：

```text
調查有哪些免費匯率 API 可以使用，整理其限制、更新頻率、授權與呼叫上限。
```

這類任務的產出通常是比較表、筆記、研究結論或風險整理。

------

### 2. 決策任務

用來做出選擇。

例如：

```text
決定匯率資料更新策略：即時查詢、定時同步，或混合模式。
```

這類任務的產出通常是架構決策、方案選擇或取捨紀錄。

------

### 3. 設計任務

用來產生規格、模型或設計。

例如：

```text
設計匯率查詢 API contract，包含 request、response、錯誤碼與範例。
```

這類任務的產出通常是 API 文件、資料模型、系統邊界、流程圖或規格文件。

------

### 4. 實作任務

用來改變系統或產出可運作成果。

例如：

```text
實作 GET /exchange-rates API，支援 baseCurrency、targetCurrency 與 date 查詢參數。
```

這類任務的產出通常是程式碼、設定檔、測試、部署結果或可執行功能。

------

## 八、RonFlow 應支援的工作流概念

RonFlow 需要把工作流程分成兩大階段：

```text
任務成形階段
任務執行階段
```

### 1. 任務成形階段

這個階段處理的是還不夠清楚的工作。

重點不是馬上執行，而是讓任務逐漸變得可以執行。

此階段關注：

```text
問題定義
範圍釐清
未知辨識
依賴整理
任務拆解
方案決策
產出定義
完成條件定義
```

------

### 2. 任務執行階段

這個階段處理的是已經足夠清楚的任務。

重點是安排、執行、追蹤與驗收。

此階段關注：

```text
待辦
進行中
審查
完成
驗收
回顧
```

------

## 九、以分散式匯率查詢系統為例

原始任務：

```text
設計一個分散式匯率查詢系統
```

RonFlow 應該判斷這不是一個可直接執行的任務，而是一個大型抽象工作。

原因包括：

```text
範圍過大
完成條件不明
架構決策未定
外部依賴未定
資料一致性需求未定
快取與容錯策略未定
```

RonFlow 應該協助使用者將它拆解為多個工作方向：

```text
釐清需求
設計架構
實作核心查詢
實作資料同步
實作快取與容錯
驗證與監控
```

其中「釐清需求」可以再拆解為：

```text
定義使用者查詢場景
定義支援幣別範圍
定義資料即時性需求
定義查詢量預估
定義成功條件
```

「設計架構」可以再拆解為：

```text
決定服務邊界
決定資料儲存方式
決定匯率來源 Provider
決定快取策略
決定排程同步策略
決定失敗降級策略
```

其中某些項目仍然不是實作任務，而是調查任務或決策任務。

例如：

```text
決定快取策略
```

可以進一步拆成：

```text
調查 Redis cache aside pattern
定義匯率查詢 cache key 格式
決定 TTL
決定外部 API 失敗時是否回傳 stale rate
撰寫快取策略決策紀錄
```

最後，真正適合進入執行流程的任務會變成：

```text
撰寫匯率查詢 API contract，包含 baseCurrency、targetCurrency、date 三個參數，以及成功、查無資料、外部服務失敗三種 response 範例。

建立 ExchangeRates 資料表 migration，欄位包含 BaseCurrency、TargetCurrency、Rate、RateDate、Source、CreatedAt。

實作 ExchangeRateProvider 介面，定義 FetchLatestRatesAsync(baseCurrency) 方法。

實作 Redis 快取 key：exchange-rate:{base}:{target}:{date}，並設定預設 TTL 為 1 小時。
```

這些任務具備明確產出、完成條件與執行方向，因此可以進入 RonFlow 既有的任務執行流程。

------

## 十、RonFlow 的價值主張

RonFlow 的價值不只是幫使用者記錄待辦事項，而是幫助使用者管理工作從模糊到清楚、從抽象到具體、從想法到執行的過程。

RonFlow 應該能夠幫助使用者避免以下問題：

```text
把過大的任務直接放進 Todo
把尚未決策的事情誤認為可以實作
把調查任務和實作任務混在一起
不知道任務為什麼卡住
不知道下一步該拆什麼
不知道任務是否已經足夠小
```

RonFlow 應該讓使用者能清楚知道：

```text
這個工作目前還有什麼未知？
這個工作需要先做什麼決策？
這個工作能不能再拆？
哪些子任務已經可以執行？
哪些子任務還只是調查或設計？
整個大型工作目前成熟到什麼程度？
```

------

## 十二、成功判準

如果 RonFlow 成功支援這類需求，使用者應該能做到以下事情：

```text
1. 可以放心地把大型、模糊、抽象的工作放進 RonFlow。
2. 可以看見大型工作被拆解後的結構。
3. 可以知道每個工作項目目前是否已經足夠清楚。
4. 可以區分調查、決策、設計與實作任務。
5. 可以知道某個任務為什麼還不能執行。
6. 可以把尚未成形的工作逐步轉化為可執行任務。
7. 可以讓已經 Ready 的任務自然進入既有執行流程。
8. 可以追蹤一個大型工作從模糊目標到完成交付的完整歷程。
```

最終，RonFlow 應該讓使用者不只是完成任務，而是更有系統地讓任務「成形」。

------

## 十三、總結

RonFlow 原本處理的是任務的執行問題。

但在實際工作中，許多重要工作一開始並不是任務，而是模糊目標、抽象問題或大型工作方向。

因此，RonFlow 需要新增一種能力：管理任務成形的過程。

可以將 RonFlow 的新方向總結為：

```text
RonFlow 幫助使用者把尚未成形的工作，逐步轉化成可以執行、可以驗收、可以追蹤的任務。
```

也可以用更簡短的一句話描述：

```text
RonFlow 的前半段負責讓任務變清楚，後半段負責讓任務被完成。
```

這將使 RonFlow 從單純的任務流程管理工具，進一步成為一套工作釐清、任務分解與執行追蹤系統。
