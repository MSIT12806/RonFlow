# RonFlow Hatchery 訪談一：問題清單與提問設計說明

## 文件目的

本文件整理「Hatchery 訪談一：問題與使用情境收斂」中實際使用的訪談問題，並補充每一道問題的設計目的、背後思考點，以及希望透過該問題收斂出的需求資訊。

訪談一的核心目的不是定義畫面、資料表或 API，而是先確認：

- 使用者在什麼情境下會遇到大型、抽象、尚未成形的任務。
- 這些任務為什麼無法直接進入 RonFlow 既有 Flow。
- 使用者目前如何在 RonFlow 外部處理這些任務。
- RonFlow 若要支援這類任務，第一階段最需要補上的能力是什麼。
- MVP 應該包含哪些範圍，以及哪些項目應明確排除。

---

# 一、訪談背景

RonFlow 原本主要管理的是已經可以執行的任務，也就是使用者大致知道要做什麼、完成後會留下什麼、如何判斷完成的工作。

但使用者實際上常會遇到另一類工作：

```text
大型、抽象、模糊、尚未被拆解、尚未能直接執行的工作。
```

例如：

```text
設計一個分散式匯率查詢系統
改善工程團隊產出衡量方式
規劃一個新產品功能
從零開始經營 YouTube 頻道
建立 RonFlow Hatchery 功能本身
```

這類工作若直接放進 Todo / Active / Review / Done，容易出現以下問題：

- 任務過大，不知道第一步要做什麼。
- 完成條件不明確，不知道做到什麼程度算完成。
- 任務中包含太多未知、決策與設計。
- 父任務與子任務的完成關係不清楚。
- 任務可能一直卡在 Flow 裡，但實際上沒有有效推進。

因此，訪談一的目標是先收斂問題場景，而不是直接進入解法設計。

---

# 二、訪談一問題清單與設計說明

## 第 1 組：目前卡住的真實場景

### Q1. 你現在使用 RonFlow 時，最常遇到哪一類「放進去也推不動」的大任務？

#### 原始問題

請列出 3 到 5 個你真的可能會放進 RonFlow 的大型／抽象任務。

例如：

```text
設計一個分散式匯率查詢系統
規劃一個新產品功能
改善工程團隊產出衡量方式
從零開始經營 YouTube 頻道
建立一套會員系統
```

#### 為什麼問這題

這題用來取得真實使用場景，避免後續討論停留在抽象概念。

如果沒有具體任務案例，後續很容易直接跳到功能想像，例如「要不要有樹狀結構」、「要不要有 AI 拆解」、「要不要有看板」，但無法判斷這些功能是否真的對應到使用者的痛點。

#### 思考點

我想確認的是：

- 哪些任務真的會被使用者拿來放進 RonFlow。
- 這些任務的共同特徵是什麼。
- 它們是專案、目標、工作項目，還是任務。
- 它們為什麼不像一般 Todo 一樣可以直接執行。

#### 預期收斂

這題會幫助定義 Hatchery 的主要需求類型：

```text
探索型需求
任務成形需求
大型任務拆解需求
```

---

### Q2. 當這些大任務放進 RonFlow 後，你通常是怎麼卡住的？

#### 原始問題

可以從下面選，也可以補充：

```text
A. 不知道第一步要做什麼
B. 不知道要拆成哪些子任務
C. 不知道哪些是調查、哪些是決策、哪些是實作
D. 不知道任務做到什麼程度算完成
E. 不知道哪些未知問題要先釐清
F. 不知道是否已經可以進入 Todo / Active
G. 任務在流程上有移動，但實際上沒有進展
H. 其他
```

#### 為什麼問這題

Q1 取得「任務類型」，Q2 則要取得「卡住的機制」。

同樣是大型任務，卡住原因可能不同：

- 有些是因為不知道怎麼開始。
- 有些是因為不知道怎麼拆。
- 有些是因為完成條件不明。
- 有些是因為它其實還是調查或決策，不是實作任務。
- 有些是因為被錯誤放進 Flow，造成 Active 區域污染。

不同卡點會導致不同產品設計。

#### 思考點

我想判斷 Hatchery 的重點到底是：

```text
幫助使用者開始？
幫助使用者拆解？
幫助使用者分類？
幫助使用者定義完成？
幫助使用者判斷 Ready？
```

使用者回答後，核心痛點被收斂為：

```text
大型任務很難拆成子任務；
子任務本身也可能缺乏具體完成概念；
因此任務難以被完成。
```

#### 預期收斂

這題支援後續提出：

```text
遞迴拆解
Ready checklist
父任務與葉節點任務區分
任務成形階段與任務執行階段分離
```

---

### Q3. 你現在如果不用 RonFlow 管理這類任務，通常會怎麼處理？

#### 原始問題

例如：

```text
寫在 Markdown
丟給 ChatGPT 討論
放在 Todo
自己腦中想一想
畫流程圖
直接開始寫程式
拆成很多 checklist
放著不處理
```

#### 為什麼問這題

這題用來找出現行替代方案，也就是使用者現在如何繞過 RonFlow 的限制。

如果使用者已經有一個外部工作流，代表 RonFlow 缺少的功能可能不是全新需求，而是目前被其他工具暫時承擔。

#### 思考點

我想確認：

- RonFlow 現在的邊界在哪裡。
- 使用者在哪些階段離開 RonFlow。
- 外部工具承擔的是「規劃」、「拆解」、「記錄」，還是「決策」。
- 哪些外部處理流程應該被整合回 RonFlow。

使用者回答後，可以確認：

```text
Markdown 目前是 RonFlow 的前置規劃區。
使用者會先在檔案系統中整理任務，
直到任務足夠細緻後，才建立到 RonFlow。
```

#### 預期收斂

這題支援後續提出：

```text
RonFlow 需要補上任務成形階段。
```

也就是：

```text
以前：Markdown 負責任務成形，RonFlow 負責任務執行。
未來：RonFlow 同時負責任務成形與任務執行。
```

---

### Q4. 你希望 RonFlow 在這類任務上，幫你做到哪一種程度？

#### 原始問題

請用直覺回答：

```text
A. 幫我記錄大型任務與子任務結構
B. 幫我判斷任務是否還太模糊
C. 幫我提示：下一步應該先調查、決策、設計，還是實作
D. 幫我產生拆解建議
E. 幫我追蹤未知問題與待決策事項
F. 幫我判斷哪些任務已經 Ready，可以進入既有流程
G. 幫我把整個大型任務從模糊到完成的歷程留下來
H. 其他
```

#### 為什麼問這題

這題用來界定 RonFlow 在 Hatchery 階段的責任範圍。

一個產品可以只做結構記錄，也可以做檢核、提醒、拆解建議、AI 輔助、決策追蹤、歷程管理。這些範圍差異很大，必須先確認使用者真正期待的層級。

#### 思考點

我想判斷：

- 使用者只是想要任務樹，還是需要成熟度管理。
- 是否需要系統判斷任務太大。
- 是否需要自動拆解或 AI 介入。
- Ready 任務是否要能接回既有 Flow。
- 是否需要留下大型任務從模糊到完成的歷程。

使用者回答後，需求被收斂為三點：

```text
1. 大型任務可以放心放進 RonFlow。
2. RonFlow 可以遞迴拆分子任務。
3. RonFlow 可以透過機制判斷任務是否太大，以及是否足夠小到進入 Flow。
```

#### 預期收斂

這題是 MVP 範圍的基礎。

初步 MVP 核心能力變成：

```text
任務樹
遞迴拆解
Ready 檢核
Ready 任務送入既有 Flow
```

---

### Q5. 對你來說，這個功能如果做對了，你會看到什麼明顯改善？

#### 原始問題

請用這種格式回答：

```text
以前我會＿＿＿＿＿＿。
現在 RonFlow 可以讓我＿＿＿＿＿＿。
所以我能＿＿＿＿＿＿。
```

#### 為什麼問這題

這題用來定義成功判準。

功能需求常常會越談越多，因此需要回到「使用者行為是否真的改變」。如果使用者能說出以前和未來的差異，就能形成可驗收的價值敘述。

#### 思考點

我想確認：

- 成功不是「系統多了一個樹狀結構」。
- 成功也不是「可以建立很多子任務」。
- 成功應該是使用者不用在 RonFlow 外面另開一套前置規劃流程。
- 使用者可以直接用 RonFlow 管理各種大小的任務。

使用者回答後，成功判準被收斂為：

```text
以前使用者需要先判斷任務是否能放進 RonFlow，
如果太大，就要先在 Markdown 中整理。

未來使用者可以直接把各種大小的任務放進 RonFlow，
並在系統內完成規劃、拆解與分配。
```

#### 預期收斂

這題支援後續寫成核心需求命題：

```text
RonFlow 需要新增一套任務成形機制，使使用者能放心建立大型、抽象、尚未規劃完成的工作，並透過遞迴拆解、成熟度判斷與 Ready 檢查，逐步將其轉化為可進入既有流程的可執行任務。
```

## 第 2 組：大型任務在系統中的概念與結構

### Q6. 你希望大型任務剛被建立時，它在 RonFlow 裡應該叫什麼？

#### 原始問題

例如：

```text
大型任務
抽象任務
目標
工作項目
專案
Epic
Hatchery Item
尚未成形任務
```

也可以選擇統一都叫「任務」，只是任務有不同成熟度。

#### 為什麼問這題

這題用來釐清 Domain Language。

如果一開始沒有釐清名詞，後續模型會混亂，例如：

```text
大型任務和專案是不是同一件事？
目標什麼時候變成任務？
父任務和子任務是否都叫任務？
```

#### 思考點

我想確認：

- 使用者心理上如何稱呼尚未成形的工作。
- 是否需要引入「目標」這個概念。
- 是否會因為命名不同，造成系統界線更複雜。
- 是否應該為了簡化模型，暫時全部稱為任務。

使用者回答後，出現一個重要問題：

```text
目標下面可以切子目標，
但到什麼時候可以改叫任務？
如果界線不清楚，反而會造成系統複雜。
```

因此初步傾向：

```text
系統可以先統一叫任務，
再用成熟度與結構描述它是否可執行。
```

#### 預期收斂

這題支援後續第二次訪談的 Domain Glossary 討論。

---

### Q7. 你認為「大型任務」跟「專案」要不要分開？

#### 原始問題

例如：

```text
設計一個分散式匯率查詢系統
```

它比較像：

```text
A. 一個很大的任務
B. 一個專案
C. 一個目標
D. 一個尚未成形的工作
E. 看情況，不需要嚴格區分
```

#### 為什麼問這題

這題用來釐清 RonFlow 的頂層結構。

如果專案和大型任務不分，系統容易把 Project 當成一個超大型 Task，後續在完成狀態、統計、Flow 接軌上都會混亂。

#### 思考點

我想確認：

- Project 是不是 Task Tree 的根節點。
- Project 是不是也會進 Flow。
- Project 是否有完成狀態。
- Project 是工作區域，還是工作本身。

使用者明確回答：

```text
要分開，專案是一個工作區域，概念上是不一樣的。
```

#### 預期收斂

因此模型初步變成：

```text
Project = 工作區域 / 容器 / 脈絡
Task Tree = 專案內的任務拆解結構
Task = 任務樹上的節點
```

---

### Q8. 當一個任務還太大、不能執行時，你希望 RonFlow 怎麼提醒你？

#### 原始問題

例如：

```text
A. 標示為「尚未 Ready」
B. 顯示「不可執行原因」
C. 顯示「建議繼續拆解」
D. 問你一組檢查問題
E. 讓 AI 提出拆解建議
F. 不阻擋，只是提醒
G. 阻擋它進入 Active
```

#### 為什麼問這題

這題用來釐清「尚未成形任務」與既有 Flow 的關係。

一個重要問題是：

```text
大型任務是否應該出現在 Todo / Active / Review / Done？
```

如果大型任務也能進 Active，它可能長期卡住，造成使用者壓力與看板污染。

#### 思考點

我想確認：

- 大型任務是否應該走 Flow。
- 不 Ready 的任務應停留在哪裡。
- 系統是要阻擋，還是提醒。
- 任務太大的判斷應透過什麼機制完成。

使用者回答後，關鍵共識是：

```text
需要持續規劃、分解的任務，本身就不應該走 Flow。
它的完成定義不清楚，放進 Active 會礙眼。
任務太大時，應該透過一系列問題去檢核。
```

#### 預期收斂

這題導出核心產品分層：

```text
Hatchery / 任務樹：負責任務成形
Flow：只負責已經 Ready 的可執行任務
```

---

### Q9. 你希望 RonFlow 判斷「任務足夠小」時，是用嚴格規則，還是用輔助提醒？

#### 原始問題

例如：

```text
A. 嚴格規則：不符合 Ready checklist 就不能進入 Active
B. 輔助提醒：系統提示可能太大，但使用者仍可強制執行
C. 混合模式：預設提醒，但某些流程可以設定成強制
```

#### 為什麼問這題

這題用來確認系統控制程度。

如果 Ready 是硬性規則，系統會更嚴謹，但也可能讓使用者覺得麻煩。  
如果 Ready 是輔助提醒，系統較彈性，但可能失去流程約束力。

#### 思考點

我想確認：

- RonFlow 是強制流程工具，還是輔助管理工具。
- 使用者是否需要 override。
- Ready checklist 是否應當成 gating rule。
- 系統是否要導入 AI 判讀。

使用者回答後，得到兩個重要限制：

```text
1. Ready 判斷先採輔助提醒。
2. RonFlow 不導入 AI 輔助判讀，所有判斷應由明確程式碼實作。
```

#### 預期收斂

這題支援後續 MVP 限制：

```text
不做 AI 自動拆解
不做 AI Ready 判讀
使用者填寫欄位與 checklist
系統根據明確規則提示狀態
```

---

### Q10. 如果 RonFlow 支援遞迴拆解，你希望看到的是哪種結構？

#### 原始問題

例如：

```text
A. 樹狀結構：任務底下有子任務，子任務底下還能有子任務
B. 大綱結構：像 Markdown outline 一樣展開收合
C. Board 結構：每一層任務都能在看板上流動
D. Mind map 結構：像任務分解圖
E. 混合：大綱拆解 + Ready 任務進入看板
```

#### 為什麼問這題

這題用來確認任務拆解的資料結構與主要互動方式。

使用者現在用 Markdown 管理，所以大綱式操作可能更接近既有習慣。  
但若使用者偏好視覺化，也可能需要 Mind Map。  
如果每層任務都能在 Board 流動，模型會變得更複雜。

#### 思考點

我想確認：

- 任務拆解需要幾層。
- 子任務是否還能再拆。
- 使用者偏好的呈現方式是文字、大綱、看板還是圖形。
- MVP 是否需要視覺化。

使用者回答後，關鍵共識是：

```text
底層是樹狀結構。
大綱結構與 Mind Map 本質上可以共用同一個資料模型，只是呈現方式不同。
```

#### 預期收斂

因此 MVP 初步方向是：

```text
底層資料模型：Task Tree
MVP 呈現方式：大綱式樹狀結構
未來可擴充：Mind Map View
```

## 第 3 組：完成定義、Ready 判斷、MVP 邊界

### Q11. 一個「父任務」什麼時候算完成？

#### 原始問題

假設有一個任務：

```text
設計分散式匯率查詢系統
```

下面拆出：

```text
調查匯率 API
決定資料更新策略
設計 API contract
實作查詢 API
實作快取
撰寫測試
```

那父任務什麼時候算完成？

候選答案：

```text
A. 所有子任務完成，父任務自動完成
B. 父任務只是容器，不需要完成狀態
C. 父任務有自己的完成條件，和子任務完成狀態分開
D. 父任務可以標記為「拆解完成」，但不等於整件事完成
E. 看任務類型決定
```

#### 為什麼問這題

這題用來釐清父任務的狀態語意。

如果父任務只是容器，它不需要完成狀態。  
但如果未來要做完成時間、複雜度、統計，那父任務可能也需要自己的完成記錄。

#### 思考點

我想確認：

- 父任務是純結構節點，還是也代表一個可追蹤工作。
- 父任務完成是否等於所有子任務完成。
- 父任務是否需要「拆解完成」這種中間狀態。
- 未來統計是否會依賴父任務狀態。

使用者回答後，得到重要共識：

```text
父任務可以標記為拆解完成。
所有子任務完成後，父任務也要有完成節點，以利後續統計。
```

#### 預期收斂

父任務需要至少兩種概念：

```text
拆解完成
任務完成
```

---

### Q12. 「拆解完成」和「任務完成」要不要分開？

#### 原始問題

我目前懷疑 RonFlow 需要分成兩種完成概念：

```text
拆解完成：這個大型任務已經被拆到足夠清楚
任務完成：實際工作已經完成
```

這兩個概念要不要明確分開？

#### 為什麼問這題

這題延續 Q11，用來確認狀態模型是否需要同時管理「成形」與「完成」。

很多任務管理工具只有 Done，但 Hatchery 的特殊性在於：

```text
一個任務可能已經拆解完成，但它底下的實作還沒完成。
```

#### 思考點

我想確認：

- RonFlow 是否需要同時管理 Shaping status 與 Execution status。
- 拆解完成是否可以成為父任務的狀態。
- 拆解完成是否不等於 Done。

使用者回答：

```text
要分開。
```

#### 預期收斂

這題奠定第二次訪談的核心：

```text
任務成熟度狀態
Flow 執行狀態
兩者應明確分開
```

### Q13. Ready 的判斷應該套用在哪裡？

#### 原始問題

你希望 RonFlow 檢查「是否足夠小、是否可以執行」時，是檢查：

```text
A. 每一個任務節點都可以檢查 Ready
B. 只有葉節點需要檢查 Ready
C. 只有準備送進 Flow 的任務才檢查 Ready
D. 父任務檢查「是否拆解完成」，葉節點檢查「是否 Ready」
```

#### 為什麼問這題

這題用來確認父任務與葉節點任務的規則是否不同。

如果父任務也做 Ready 檢查，容易出現矛盾：

```text
父任務有子任務，代表它還是工作包。
它是否真的應該進 Flow？
```

#### 思考點

我想確認：

- Ready 是否只屬於葉節點。
- 父任務是否應該看拆解完成，而不是 Ready。
- Flow 是否只接收可執行的葉節點任務。

使用者選擇：

```text
D. 父任務檢查「是否拆解完成」，葉節點檢查「是否 Ready」。
```

#### 預期收斂

這題導出重要規則：

```text
父任務管理拆解狀態。
葉節點管理 Ready 狀態。
Ready 葉節點才能進 Flow。
```

### Q14. 你認為一個任務進入 Flow 前，最低限度要回答哪些問題？

#### 原始問題

原本候選：

```text
1. 這個任務的動作是什麼？
2. 這個任務完成後會留下什麼產出？
3. 怎樣算完成？
4. 第一個執行步驟是什麼？
5. 是否還有未決策事項？
6. 是否還有未釐清依賴？
7. 是否可以在合理時間內完成？
```

#### 為什麼問這題

這題用來收斂 Ready checklist。

Ready checklist 是 Hatchery 的核心機制之一，但不能設計得太重。  
如果問題太多，使用者會覺得系統囉嗦。  
如果問題太少，又無法幫助判斷任務是否可執行。

#### 思考點

我想確認：

- 什麼資訊是任務進 Flow 前真正必要的。
- 哪些問題太抽象。
- 哪些問題暫時不適合 MVP。
- 「時間」應如何描述才有可操作性。

使用者回饋：

```text
1. 「這個任務的動作是什麼」有點抽象，建議拿掉。
4. 「第一個執行步驟」沒必要。
5. 未決策事項目前想像不到為什麼要問，先拿掉。
6. 未釐清依賴目前想像不到為什麼要問，先拿掉。
7. 建議改成：是否可以明確預估預計完成時間。
```

#### 預期收斂

MVP Ready checklist 初步簡化為：

```text
1. 是否有明確產出物？
2. 是否有完成條件？
3. 是否可以明確預估預計完成時間？
```

### Q15. 任務如果被判斷為「不 Ready」，RonFlow 應該讓它變成什麼？

#### 原始問題

例如：

```text
A. 保持在任務樹中，不進 Flow
B. 自動要求繼續拆解
C. 標記不可執行原因
D. 轉成調查任務
E. 轉成決策任務
F. 轉成設計任務
G. 使用者自己決定下一步
```

#### 為什麼問這題

這題用來確認不 Ready 任務的處理方式。

不 Ready 不代表錯誤，也不代表要自動拆解。  
它可能只是還在任務樹中等待進一步整理。

#### 思考點

我想確認：

- 不 Ready 是否是一種狀態，而不是錯誤。
- 系統是否要自動幫使用者做轉換。
- 是否需要自動產生調查／決策任務。
- 不 Ready 任務是否能繼續存在於 Hatchery 中。

使用者回答：

```text
A. 保持在任務樹中，不進 Flow。
```

#### 預期收斂

這題支援 MVP 規則：

```text
不 Ready 任務不進 Flow。
它保留在任務樹中，等待使用者繼續拆解或補充資訊。
```

### Q16. 你需要「任務類型」嗎？

#### 原始問題

例如每個節點可以標記：

```text
目標
調查
決策
設計
實作
驗收
文件
溝通
```

或只分：

```text
尚未成形
可執行任務
```

#### 為什麼問這題

這題用來確認分類複雜度。

任務類型可以幫助分析，但也會增加輸入成本。  
如果分類過多，使用者每建立一個任務都要先想它是哪一類，反而會降低使用意願。

#### 思考點

我想確認：

- 使用者是否真的需要調查／決策／設計／實作分類。
- MVP 是否應避免過多 metadata。
- 是否可以先用 Ready / Not Ready 取代複雜分類。

使用者回答：

```text
少一點。
```

#### 預期收斂

MVP 先不強制做任務類型分類。  
調查、決策、設計等工作可以先透過任務名稱與完成條件表示。

### Q17. 「遞迴拆解」需要支援到多深？

#### 原始問題

你比較傾向：

```text
A. 不限制層數
B. 最多 3 層
C. 最多 5 層
D. 不硬限制，但層數太深時提醒使用者
```

#### 為什麼問這題

這題用來確認系統結構限制。

如果限制層數，實作和 UI 可能比較簡單，但可能不符合使用者真實思考方式。  
如果不限制層數，資料模型要支援任意深度，UI 也要處理巢狀展示。

#### 思考點

我想確認：

- 使用者是否期待真正的遞迴任務拆解。
- 任務樹是否只是兩層父子任務，還是任意深度。
- MVP 是否需要對層級做限制。

使用者回答：

```text
不限層數。
```

#### 預期收斂

MVP 資料模型應支援任意深度任務樹。  
介面可以先以大綱方式呈現，避免一開始做複雜視覺化。

### Q18. MVP 最小版本，你認為一定要有什麼？

候選功能：

```text

3. 子任務可以繼續建立子任務
4. 任務可以標記是否 Ready
```

# 三、第一次訪談後形成的初步結論

## 1. 核心問題

RonFlow 目前只能很好地管理已經成形、可以執行的任務。  
但使用者在真實工作中，經常會遇到大型、抽象、尚未成形的工作。

這些工作若直接放進 Flow，會產生完成定義不清、無法估時、無法拆解、長期卡住、看板污染等問題。

---

## 2. 核心需求

RonFlow 需要新增一個任務成形機制，使使用者可以：

```text
放心建立大型任務
在系統內遞迴拆解子任務
透過檢核表單確認任務是否 Ready
將 Ready 任務送進既有 Flow
讓不 Ready 任務留在任務樹中繼續成形
```

---

## 3. 初步模型

```text
Project
  └── Task Tree
        ├── Parent Task
        │     ├── Child Task
        │     └── Ready Leaf Task → Flow
        └── Parent Task
```

其中：

```text
Project = 工作區域，不是任務。
Task Tree = 專案中的任務拆解結構。
Parent Task = 可被拆解的父任務。
Leaf Task = 沒有子任務的任務。
Ready Leaf Task = 通過 Ready checklist，可以進 Flow 的任務。
Flow = 既有 Todo / Active / Review / Done 執行流程。
```

---

## 4. 初步 Ready checklist

MVP 版本暫定為：

```text
1. 是否有明確產出物？
2. 是否有完成條件？
3. 是否可以明確預估預計完成時間？
```

此 checklist 後續仍可調整。

---

## 5. MVP 範圍

MVP 應包含：

```text
1. 在專案中建立任務樹。
2. 任務可以建立子任務。
3. 子任務可以繼續建立子任務，不限制層數。
4. 葉節點任務可填寫 Ready checklist。
5. 通過 Ready checklist 的任務可送入既有 Flow。
6. 不 Ready 任務保留在任務樹中。
7. 父任務可標記為拆解完成。
8. 父任務與子任務完成狀態可支援後續統計。
```

---

## 6. MVP 非目標

第一版明確不做：

```text
AI 自動拆解
AI Ready 判讀
Mind Map 視覺化
報表統計
跨專案依賴
多人協作
自動估時
複雜甘特圖
任務難度自動判斷
```

---

# 四、這些問題背後的整體訪談策略

第一次訪談採用的不是「直接問功能」的方式，而是依照以下順序逐步收斂：

```text
1. 先問真實場景
2. 再問卡住方式
3. 再問目前替代方案
4. 再問 RonFlow 應補上的能力
5. 再問成功後的行為差異
6. 再釐清核心名詞與系統邊界
7. 再釐清任務樹與 Flow 的關係
8. 再收斂 Ready 判斷
9. 最後切出 MVP 與非目標
```

這樣做的原因是：

```text
避免一開始就掉進畫面或技術設計；
避免把大型任務誤當成一般 Todo；
避免過早引入 AI、報表、視覺化等非核心功能；
先確認 RonFlow 真正要解決的是「任務成形」問題。
```

# 五、後續訪談方向

第一次訪談完成後，第二次訪談應進入：

```text
Hatchery 訪談二：任務成形模型
```

重點應包含：

```text
1. Domain glossary
2. 任務成熟度狀態
3. 父任務與葉節點規則
4. Ready checklist 明確欄位
5. 不 Ready 原因清單
6. 任務進入 Flow 的條件
7. 狀態轉換規則
```

其中最需要繼續釐清的是：

```text
1. 成熟度狀態與 Flow 狀態是否分開。
2. Ready checklist 的欄位是否足夠。
3. 父任務完成與子任務完成的關係。
4. 已 Ready 任務新增子任務時，狀態如何變化。
5. 不 Ready 原因是使用者選擇，還是由欄位缺漏自動推導。
```
