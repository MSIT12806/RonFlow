# RonFlow Hatchery 訪談二：問題清單與提問設計說明

## 文件目的

本文件整理「Hatchery 訪談二：任務成形模型」中實際使用的訪談問題，並補充每一道問題的設計目的、背後思考點，以及希望透過該問題收斂出的需求資訊。

訪談二的核心目的，是把第一次訪談中確認的「大型任務需要在 RonFlow 內部被拆解與成形」進一步轉換成可建模的規則。

也就是說，訪談二不處理畫面細節、API、資料表或前後端實作，而是先定義：

- RonFlow 裡有哪些核心名詞。
- 任務在 Hatchery 階段有哪些成熟度狀態。
- 父任務與葉任務的規則有何不同。
- 什麼任務可以進入既有 Flow。
- Ready checklist 應包含哪些條件。
- 不 Ready 任務要如何處理。
- 任務成熟度狀態如何轉換。

---

# 一、訪談背景

第一次訪談已確認：

```text
RonFlow 不只需要管理已經可以執行的任務，
也需要管理尚未成形的大型、抽象、模糊工作。
```

第一次訪談後形成的核心共識是：

```text
Project 是工作區域，不是任務。
大型任務不應直接進入 Flow。
RonFlow 需要支援任務樹與遞迴拆解。
只有通過 Ready 檢核的葉任務，才適合進入既有 Flow。
MVP 不導入 AI 輔助判讀，所有判斷應以明確規則與使用者填寫為主。
```

因此，第二次訪談的重點是定義任務成形模型：

```text
Hatchery 如何描述一個任務尚未成形？
一個任務如何從模糊逐步變成 Ready？
父任務與葉任務的狀態如何區分？
Ready 任務如何接回既有 Flow？
```

# 二、訪談二問題清單與設計說明

## 第 1 組：Domain Glossary

### Q21. RonFlow 裡的核心名詞，你希望怎麼命名？

#### 原始問題

我先提出一組候選：

```text
Project：專案／工作區
Task：任務節點
Parent Task：父任務
Child Task：子任務
Leaf Task：葉節點任務，沒有子任務的任務
Task Tree：任務樹
Ready Task：可執行任務
Flow Task：已進入既有流程的任務
Split Complete：拆解完成
Done：任務完成
```

你比較希望中文介面怎麼叫？

尤其這幾個要不要改名：

```text
任務節點
葉節點任務
拆解完成
可執行任務
已進入流程
```

#### 為什麼問這題

這題用來建立 Hatchery 的共通語言。

第一次訪談已經發現「目標」、「任務」、「專案」、「工作項目」之間容易混淆。如果名詞沒有先定義，後續討論狀態、規則與畫面時會很容易發散。

尤其是「葉節點任務」這種詞，雖然技術上準確，但介面上可能太生硬，需要確認使用者能否接受。

#### 思考點

我想確認：

- 使用者是否接受所有節點都叫任務。
- 「葉節點任務」在中文介面上是否需要簡化。
- 「Task」是否應翻成任務，而不是工作項目、目標或事項。
- 後續文件、畫面與資料模型要使用哪一組語彙。

#### 使用者回答後的收斂

使用者確認：

```text
Task -> 任務
葉節點任務 -> 葉任務
```

#### 預期收斂結果

後續可採用以下基本名詞：

```text
任務
父任務
子任務
葉任務
任務樹
Ready
Flow
拆解完成
任務完成
```

### Q22. 「任務節點」這個概念你接受嗎？

#### 原始問題

因為在任務樹裡，每一層都叫「任務」可能會有點怪，例如：

```text
設計 RonFlow Hatchery
  ├── 釐清任務成形模型
  ├── 設計任務樹資料結構
  └── 實作 Ready 檢核
```

這三個都叫任務，但有些其實比較像「方向」、「階段」、「工作包」。

你比較傾向：

```text
A. 全部都叫「任務」
B. 在資料模型叫 Task Node，介面上叫「任務」
C. 父層叫「工作項目」，葉節點才叫「任務」
D. 父層叫「目標」，葉節點叫「任務」
E. 先全部叫任務，不要過早拆名詞
```

#### 為什麼問這題

這題用來避免名詞過度建模。

從建模角度來看，父層任務可能更像工作包、階段或目標；葉任務才像真正可執行任務。但如果介面上區分太細，使用者每建立一個項目都要先判斷它到底是目標、工作項目還是任務，反而增加使用成本。

#### 思考點

我想確認：

- RonFlow 是否需要引入 Task Node 這種抽象概念。
- 父層與葉層是否需要不同名稱。
- 使用者是否願意接受「所有節點都是任務，只是狀態不同」。
- MVP 是否應避免過早把名詞拆太細。

#### 使用者回答後的收斂

使用者選擇：

```text
A. 全部都叫「任務」
```

#### 預期收斂結果

MVP 中，所有任務樹節點介面上都稱為「任務」。  
父任務與葉任務不是靠名稱區分，而是靠結構區分：

```text
有子任務 = 父任務
沒有子任務 = 葉任務
```

## 第 2 組：任務成熟度狀態

### Q23. 你認為任務成熟度需要幾個狀態？

#### 原始問題

我先列一版：

```text
1. 未整理：剛建立，還只是想法或大方向
2. 拆解中：正在被拆成子任務
3. 拆解完成：父任務已經拆到可接受程度
4. 可執行：這個任務通過 Ready checklist，可以進 Flow
5. 已進入 Flow：已被送到既有 Todo / Active / Review / Done
6. 已完成：任務本身完成
```

你覺得這樣太多、太少，還是方向正確？

#### 為什麼問這題

第一次訪談已確認 RonFlow 不能只管理 Todo / Active / Review / Done，還需要管理「任務是否成形」。這題用來提出第一版成熟度狀態，讓使用者判斷狀態數量是否足夠。

#### 思考點

我想確認：

- 任務成形階段是否需要多個狀態。
- 「未整理」、「拆解中」、「拆解完成」、「Ready」是否能描述任務從模糊到可執行的過程。
- 「已進入 Flow」與「已完成」是否應納入成熟度狀態。
- 使用者是否覺得狀態太多，造成操作負擔。

#### 使用者回答後的收斂

使用者表示：

```text
好像差不多，可以暫定這樣。
```

#### 預期收斂結果

此狀態集可作為後續狀態轉換表的初稿，但仍需要依父任務與葉任務拆成不同路徑。

### Q24. 「拆解完成」應該只存在於父任務嗎？

#### 原始問題

我的初步想法是：

```text
父任務可以有：拆解中 / 拆解完成
葉節點可以有：未 Ready / Ready / 已進入 Flow / 已完成
```

因為葉節點沒有子任務，所以它不需要「拆解完成」。

你同意嗎？

還是任何任務都可以標記「拆解完成」？

#### 為什麼問這題

這題用來確認父任務與葉任務是否使用不同狀態。

如果所有任務都能標記拆解完成，會出現語意混亂：葉任務沒有子任務，理論上無法「拆解完成」。反之，父任務通常不該檢查 Ready，因為它還是一個容器或工作包。

#### 思考點

我想確認：

- 拆解完成是否只屬於父任務。
- 葉任務是否只需要 Ready / Flow / Done。
- 狀態是否應根據任務是否有子任務而變化。
- UI 上是否可以根據任務結構顯示不同操作。

#### 使用者回答後的收斂

使用者同意：

```text
拆解完成只存在於父任務。
```

#### 預期收斂結果

狀態模型可拆成：

```text
父任務：拆解中 / 拆解完成 / 已完成
葉任務：未 Ready / Ready / 已進入 Flow / 已完成
```

### Q25. 一個任務可以同時有「成熟度狀態」與「Flow 狀態」嗎？

#### 原始問題

例如：

```text
成熟度狀態：Ready
Flow 狀態：Todo
```

或：

```text
成熟度狀態：已進入 Flow
Flow 狀態：Active
```

你比較傾向：

```text
A. 分開存，成熟度狀態和 Flow 狀態是兩套狀態
B. 合併成一套狀態，避免複雜
C. Hatchery 階段只看成熟度，進 Flow 後只看 Flow 狀態
D. 使用者不需要看到兩套狀態，但系統內部可以分開
```

#### 為什麼問這題

這題用來定義 Hatchery 與既有 Flow 的狀態邊界。

如果一個任務同時顯示成熟度狀態與 Flow 狀態，系統會更完整，但也可能造成使用者困惑。例如一個任務同時是 Ready、Todo、Active，使用者可能不知道哪個狀態才是主要狀態。

#### 思考點

我想確認：

- Hatchery 狀態是否只存在於任務進 Flow 前。
- 任務進入 Flow 後，是否應交給既有 Flow 管理。
- 使用者是否需要看到兩套狀態。
- 狀態合併是否會導致語意不清。

#### 使用者回答後的收斂

使用者選擇：

```text
C. Hatchery 階段只看成熟度，進 Flow 後只看 Flow 狀態。
```

#### 預期收斂結果

Hatchery 與 Flow 狀態分工如下：

```text
未進 Flow：看 Hatchery 成熟度狀態。
已進 Flow：看既有 Flow 狀態。
```

### Q26. 「已完成」應該怎麼判定？

#### 原始問題

第一次訪談你提到，父任務也要有完成節點，以利後續統計，例如完成時間、複雜度。

那父任務完成時，你比較傾向：

```text
A. 所有子任務完成後，父任務自動完成
B. 所有子任務完成後，系統提示使用者手動確認父任務完成
C. 父任務可以手動完成，不一定等所有子任務完成
D. 父任務完成 = 拆解完成 + 所有子任務完成
E. 父任務完成 = 使用者明確按下完成，子任務完成只是參考
```

#### 為什麼問這題

第一次訪談已確認父任務也要有完成節點，以利後續統計。這題用來進一步確認父任務完成是手動還是自動。

#### 思考點

我想確認：

- 父任務完成是否必須依賴子任務完成。
- 使用者是否需要手動驗收父任務。
- 父任務完成時間應如何計算。
- 是否要允許子任務未完成但父任務完成。

#### 使用者回答後的收斂

使用者選擇：

```text
A. 所有子任務完成後，父任務自動完成。
```

#### 預期收斂結果

父任務完成規則：

```text
當所有直接或遞迴子任務完成後，父任務可自動完成。
```

此規則後續需要再確認是否包含所有 descendant，還是只看 direct children。

## 第 3 組：哪些任務可以進 Flow

### Q27. 是否只有「葉節點任務」可以進 Flow？

#### 原始問題

也就是：

```text
有子任務的父任務：不能進 Flow
沒有子任務的葉節點：通過 Ready checklist 後，可以進 Flow
```

這樣可以避免父任務卡在 Active，也能讓 Flow 只處理真正可執行的工作。

你同意嗎？

#### 為什麼問這題

這題用來確認 Flow 的入口規則。

第一次訪談已發現大型任務不應直接進 Flow，否則會出現完成定義不清與 Active 區域污染。因此需要確認：是不是只有葉任務能進 Flow。

#### 思考點

我想確認：

- 父任務是否永遠不能進 Flow。
- Flow 是否只處理實際可執行的工作。
- 葉任務是否必須通過 Ready checklist 才能進 Flow。
- 系統是否可用「是否有子任務」判斷能否進 Flow。

#### 使用者回答後的收斂

使用者回答：

```text
是。
```

#### 預期收斂結果

核心規則：

```text
只有葉任務可以進 Flow。
父任務不能進 Flow。
```

### Q28. 如果一個父任務本身也有工作內容，怎麼辦？

#### 原始問題

例如：

```text
設計 Hatchery 任務成形模型
  ├── 定義成熟度狀態
  ├── 定義 Ready checklist
  └── 定義不可執行原因
```

但父任務「設計 Hatchery 任務成形模型」本身可能也需要產出一份模型文件。

你希望：

```text
A. 父任務不能自己被執行；要另外拆一個子任務「撰寫任務成形模型文件」
B. 父任務可以自己也進 Flow
C. 父任務可以有自己的完成條件，但不能進 Flow
D. 看使用者決定
```

#### 為什麼問這題

Q27 確認父任務不能進 Flow，但現實中父任務名稱本身可能也暗示一個產出物。這題用來處理這個邊界情境。

#### 思考點

我想確認：

- 父任務是否可以同時作為容器與執行項目。
- 如果父任務本身有工作內容，是否應拆成明確葉任務。
- 是否要避免父任務與葉任務雙重語意。
- Flow 是否保持只接收葉任務的單純規則。

#### 使用者回答後的收斂

使用者選擇：

```text
A. 父任務不能自己被執行；要另外拆一個子任務。
```

#### 預期收斂結果

若父任務本身也需要產出物，應新增一個葉任務承接該工作。

例如：

```text
設計 Hatchery 任務成形模型
  ├── 定義成熟度狀態
  ├── 定義 Ready checklist
  ├── 定義不可執行原因
  └── 撰寫任務成形模型文件
```

### Q29. Ready 任務送進 Flow 後，原本任務樹中的節點要怎麼處理？

#### 原始問題

例如葉節點：

```text
撰寫 Ready checklist 第一版
```

送進 Todo 後，你希望任務樹裡顯示：

```text
A. 同一個任務，只是多了一個 Flow 狀態
B. 複製一份到 Flow，任務樹保留原節點
C. 任務從任務樹移走，只存在 Flow
D. 任務樹仍顯示，但標記「已送入 Flow」
```

#### 為什麼問這題

這題用來確認任務樹與 Flow 是共用任務資料，還是複製資料。

如果複製一份到 Flow，會出現同步問題：任務名稱、完成狀態、時間統計可能不一致。  
如果移出任務樹，父任務的結構會斷裂。  
因此需要確認使用者期待。

#### 思考點

我想確認：

- Ready 任務進 Flow 後是否仍屬於原本任務樹。
- 是否要避免資料重複。
- Flow 狀態是否應直接掛在同一個任務上。
- 任務樹是否仍應顯示該任務，以便父任務統計完成狀態。

#### 使用者回答後的收斂

使用者選擇：

```text
A. 同一個任務，只是多了一個 Flow 狀態。
```

#### 預期收斂結果

任務不複製、不移出。  
同一筆任務資料可同時屬於任務樹，並在進入 Flow 後由 Flow 狀態管理。

## 第 4 組：Ready checklist

- 是否還需要加入其他條件。

#### 使用者回答後的收斂

使用者回答：

```text
是。
```

#### 預期收斂結果

MVP Ready checklist 確認為：

```text
1. 有明確產出物。
2. 有完成條件。
```

### Q31. Ready checklist 是單純勾選，還是必須填欄位？

#### 原始問題

例如：

```text
產出物：一份 Hatchery 任務成形模型文件
完成條件：包含成熟度狀態、Ready checklist、不可執行原因清單
預估時間：4 小時
```

你比較傾向：

```text
A. 只要勾選三個 checkbox
B. 每個項目都要填文字
C. 產出物、完成條件要填文字；預估時間填數字
D. 先填文字，系統再讓使用者勾選確認
```

#### 為什麼問這題

這題用來確認 Ready checklist 的資料結構。

如果只是 checkbox，系統只能知道使用者認為它 Ready，但不知道產出物、完成條件與預估時間是什麼。  
如果要求填欄位，系統就能保存可驗收資訊，但輸入成本較高。

#### 思考點

我想確認：

- Ready 是否只是狀態，還是需要具體欄位支撐。
- 產出物與完成條件是否應成為任務資料的一部分。
- 使用者是否需要先規劃，再確認。
- checklist 是否應是「確認」而不是「取代規劃」。

#### 使用者回答後的收斂

使用者選擇：

```text
D. 使用者先規劃要產出什麼，確認產出後，再 checked 起來。
```

#### 預期收斂結果

Ready checklist 應包含兩層：

```text
1. 使用者先填寫規劃欄位。
2. 使用者再勾選確認該項已具備。
```

例如：

```text
產出物欄位 + 產出物確認 checkbox
```

## 第 7 組：狀態轉換規則

### Q38. 任務成熟度狀態轉換，你覺得這版合理嗎？

#### 原始問題

單一路徑版本：

```text
未整理
  ↓ 建立子任務
拆解中
  ↓ 標記拆解完成
拆解完成
  ↓ 葉節點通過 Ready checklist
Ready
  ↓ 送進 Flow
已進入 Flow
  ↓ Flow 完成
已完成
```

但這裡有一個問題：父任務和葉節點不一定走同一條路。

所以可能要拆成兩條：

```text
父任務：
未整理 → 拆解中 → 拆解完成 → 子任務全完成 → 已完成

葉節點：
未整理 → Ready → 已進入 Flow → 已完成
```

你比較接受哪一種？

#### 為什麼問這題

前面已經確認父任務與葉任務的規則不同，因此這題用來確認狀態轉換是否也應分成兩條路。

若用單一路徑，父任務與葉任務的語意會混在一起。  
例如父任務不應該 Ready，葉任務不需要拆解完成。

#### 思考點

我想確認：

- 狀態轉換是否根據任務是否有子任務而改變。
- 新任務預設是葉任務還是父任務。
- 使用者建立子任務時，原任務的身份是否會改變。
- Ready checklist 應在什麼時候出現。

#### 使用者回答後的收斂

使用者接受分兩條路，並補充畫面想像：

```text
新建立出來的任務都視為葉任務，也就是要去輸入 Ready checklist。
除非使用者按下「建立子任務」，
此時原本的任務就會變成父任務，不用輸入 Ready checklist。
```

#### 預期收斂結果

重要規則：

```text
新任務預設為葉任務。
葉任務需要 Ready checklist。
當葉任務新增子任務時，該任務轉為父任務。
父任務不需要輸入 Ready checklist。
父任務改走拆解狀態。
```

### Q39. 任務狀態是否允許倒退？

#### 原始問題

例如：

```text
Ready → 未 Ready
拆解完成 → 拆解中

```

你希望：

```text

```

#### 為什麼問這題

任務成形過程中，需求常會改變。原本以為 Ready 的任務，後來可能發現還太大；原本拆解完成的父任務，也可能需要再拆。

這題用來確認狀態是否要支援倒退，以及是否需要事件紀錄。

#### 思考點

我想確認：

- Ready 狀態是否可被取消。
- 拆解完成是否可回到拆解中。
- MVP 是否需要記錄狀態歷程。
- 使用者是否接受先簡化、後補紀錄。

#### 使用者回答後的收斂

使用者選擇：

```text
D. MVP 先允許，之後再補紀錄。
```

#### 預期收斂結果

MVP 允許狀態倒退，但暫不實作完整狀態歷程。

後續可補：

```text
狀態變更紀錄
重新拆解紀錄
Ready 取消原因
```

### Q40. 如果已經 Ready 的任務又新增子任務，會發生什麼？

#### 原始問題

例如：

```text
撰寫 Ready checklist 第一版
```

原本已 Ready，後來你發現它其實還太大，新增了子任務。

系統應該：

```text
A. 自動取消 Ready
B. 保留 Ready，但提醒有矛盾
C. 不允許 Ready 任務新增子任務
D. 允許新增，但要使用者確認狀態改回拆解中
```

#### 為什麼問這題

這題是 Q38 和 Q39 的具體邊界案例。

如果 Ready 任務新增子任務，代表它不再是葉任務，也不再符合「葉任務才能 Ready」的規則。因此需要定義系統如何處理這個狀態轉換。

#### 思考點

我想確認：

- Ready 任務是否允許重新拆解。
- 新增子任務時，是否自動取消 Ready。
- 是否要使用者確認，避免狀態突然改變。
- 葉任務轉父任務時，Ready checklist 是否失效。

#### 使用者回答後的收斂

使用者選擇：

```text
D. 允許新增，但要使用者確認狀態改回拆解中。
```

#### 預期收斂結果

規則如下：

```text
Ready 葉任務可以新增子任務。
新增子任務前，系統應提示使用者：
一旦建立子任務，該任務會從葉任務轉為父任務，
Ready 狀態將失效，並改回拆解中。
使用者確認後才執行。
```

# 三、第二次訪談後形成的模型結論

## 1. 核心名詞

```text
Project：專案／工作區，不是任務。
Task：任務，任務樹上的所有節點都叫任務。
Parent Task：父任務，有子任務的任務。
Child Task：子任務。
Leaf Task：葉任務，沒有子任務的任務。
Task Tree：任務樹。
Ready：葉任務已符合可執行條件。
Flow：既有 Todo / Active / Review / Done 執行流程。
Split Complete：拆解完成，父任務專用狀態。
Done：完成。
```

---

## 2. 父任務與葉任務規則

```text
新建立任務預設為葉任務。
葉任務需要 Ready checklist。
葉任務通過 Ready checklist 後可以進 Flow。
父任務是有子任務的任務。
父任務不能進 Flow。
父任務不需要 Ready checklist。
父任務可以標記拆解完成。
所有子任務完成後，父任務自動完成。
```

---

## 3. Hatchery 與 Flow 的關係

```text
Hatchery 階段只看任務成熟度。
任務進入 Flow 後，只看 Flow 狀態。
任務進 Flow 後不複製資料、不移出任務樹。
同一個任務仍留在任務樹中，只是多了 Flow 狀態。
```

---

## 4. MVP Ready checklist

MVP Ready checklist 確認為：

```text
1. 有明確產出物。
2. 有完成條件。
3. 可明確預估預計完成時間。
```

操作方式：

```text
使用者先填寫產出物、完成條件、預估時間。
使用者確認後，再勾選 checklist。
三項都符合後，葉任務可視為 Ready。
```

---

## 5. 預估完成時間

```text
預估完成時間採「數字 + 單位」。
目前不設定時間門檻。
只要使用者能明確預估即可。
未來可與 Flow reminder、預計完成時間或到期時間整合。
```

---

## 6. 不 Ready 處理

MVP 不做不 Ready 原因。

```text
符合 Ready checklist = Ready。
不符合 Ready checklist = 不 Ready。
不 Ready 任務留在任務樹中。
使用者可補充 Ready 欄位，或繼續拆子任務。
```

---

## 7. 任務類型

MVP 不做任務類型分類。

```text
不做調查／決策／設計／實作標籤。
不做分類功能。
```

調查、決策、設計、實作都可以用一般任務表示，只要有產出物、完成條件與預估時間即可。

---

## 8. 狀態轉換

### 葉任務路徑

```text
新建立任務
  ↓ 填寫 Ready checklist
Ready
  ↓ 送入 Flow
進入 Flow
  ↓ Flow 完成
完成
```

### 父任務路徑

```text
新建立任務
  ↓ 建立子任務
轉為父任務／拆解中
  ↓ 使用者標記拆解完成
拆解完成
  ↓ 所有子任務完成
完成
```

### 狀態倒退

```text
MVP 允許狀態倒退。
暫不實作完整狀態歷程。
後續再補紀錄功能。
```

### Ready 任務新增子任務

```text
Ready 葉任務新增子任務時，
系統應提示使用者確認。

確認後：
Ready 狀態失效。
該任務轉為父任務。
狀態改為拆解中。
```

# 四、這些問題背後的整體訪談策略

第二次訪談的策略，是把第一次訪談得到的需求方向轉成可建模的規則。

整體順序如下：

```text
1. 先定義名詞。
2. 再區分父任務與葉任務。
3. 再定義成熟度狀態。
4. 再定義哪些任務可以進 Flow。
5. 再定義 Ready checklist。
6. 再刪除不必要的複雜功能。
7. 最後確認狀態轉換與邊界案例。
```

這樣設計的原因是：

```text
先避免名詞混亂。
再避免父任務與葉任務規則混在一起。
再避免 Hatchery 狀態與 Flow 狀態混在一起。
再確保 MVP 只保留必要機制。
最後用狀態轉換與邊界案例驗證模型是否一致。
```
