# RonFlow Hatchery 訪談紀錄一：問題與使用情境收斂

日期：2026-06-30  
訪談階段：訪談一  
訪談主題：抽象任務分解與任務成形管理  
訪談對象：User  
訪談狀態：已完成

---

## 一、訪談目的

本次訪談的目的，是確認 RonFlow 在面對大型、抽象、尚未成形的工作時，實際需要解決哪些問題。

本次不進入畫面設計、資料表設計、API 設計或技術實作細節，而是聚焦在：

- 哪些任務目前無法直接進入 RonFlow。
- 使用者現在如何處理這類任務。
- 使用者希望 RonFlow 如何承接大型任務。
- 任務何時應該繼續拆解。
- 任務何時可以進入既有 Flow。
- MVP 第一版應該包含與排除哪些內容。

---

## 二、需求類型判斷

本需求不是單純的 Todo 管理，也不是單純的子任務功能。

比較精確的定位是：

```text
探索型需求
+ 任務成形需求
+ 遞迴任務分解需求
+ Ready 檢核需求
```

RonFlow 目前主要處理的是已經足夠明確、可以被執行與驗收的任務。

但使用者實際上會遇到一些大型、抽象、尚未成形的工作，例如：

```text
設計一個分散式匯率查詢系統
規劃一個新產品功能
改善工程團隊產出衡量方式
從零開始經營 YouTube 頻道
建立一套會員系統
```

這些工作不能直接被視為可執行任務，而是需要先經過拆解、釐清、檢核，才會逐步形成可以進入 Flow 的任務。

---

## 三、目前痛點

### 1. 大型任務無法放心放進 RonFlow

目前使用者在建立任務前，會先判斷：

```text
這個任務可以直接放進 RonFlow 嗎？
還是它太大、太模糊，應該先拆成幾個子任務？
```

這會導致 RonFlow 只能管理「已經整理好的任務」，而無法管理「尚未整理完成的任務」。

使用者真正想要的是：

```text
我可以直接把大型、抽象、還沒有規劃完成的任務放進 RonFlow，
不用擔心它因為還不能執行，就一直卡在系統裡面很礙眼。
```

---

### 2. 任務拆解本身也很困難

使用者指出，大型任務之所以難處理，不只是因為它大，而是因為：

```text
大型任務很難拆成子任務，
而子任務本身也可能缺乏具體完成概念，
導致這些任務仍然難以完成。
```

因此，RonFlow 不能只支援一層子任務。

它需要支援遞迴拆解：

```text
大型任務
  ↓
子任務
  ↓
更小的子任務
  ↓
可執行任務
```

---

### 3. Markdown 目前被迫成為 RonFlow 的前置整理區

使用者目前的做法是：

```text
先用 Markdown 在檔案系統中管理大型任務，
直到任務足夠細緻，可以被建立在 RonFlow 中。
```

這個流程不方便，因為任務會被拆在兩個地方：

```text
Markdown：負責前期規劃、拆解、整理
RonFlow：負責後期執行與追蹤
```

使用者希望未來可以直接在 RonFlow 內完成：

```text
做事
規劃
拆解
分配
執行
追蹤
```

---

## 四、核心需求命題

本次訪談收斂出的核心命題如下：

```text
RonFlow 需要新增一套任務成形機制，
使使用者能放心建立大型、抽象、尚未規劃完成的工作，
並透過遞迴拆解、檢核表單與 Ready 判斷，
逐步將其轉化為可進入既有 Flow 的可執行任務。
```

更簡化地說：

```text
RonFlow 不只要管理任務如何被完成，
也要管理任務如何從模糊變清楚。
```

## 五、概念模型初步結論

### 1. 專案與任務需要分開

使用者明確指出：

```text
專案是一個工作區域，概念上和任務不同。
```

因此，RonFlow 不應該把「專案」當成超大型任務。

比較合理的概念模型是：

```text
Project
  └── Task Tree
        ├── Task
        │    ├── Subtask
        │    └── Subtask
        └── Task
```

其中：

```text
Project = 工作區域 / 容器 / 脈絡
Task Tree = 專案中的任務拆解結構
Task Node = 可以被拆解、檢核、執行或統計的工作節點
```

---

### 2. 大型任務不應直接進入既有 Flow

既有 Flow 可能是：

```text
Todo → Active → Review → Done
```

但大型、抽象、需要持續規劃的任務，本身不適合直接進入 Flow。

原因是：

```text
它的完成定義不明確。
它可能只是需要被拆解。
它可能包含很多尚未完成的子任務。
如果放進 Active，可能會長期卡住並造成視覺噪音。
```

因此，目前共識是：

```text
任務樹負責拆解與成形。
Flow 負責執行與驗收。
只有已經 Ready 的任務，才適合進入既有 Flow。
```

---

### 3. 父任務需要保留節點與統計能力

使用者認為父任務不能只是單純的視覺容器。

父任務應該保留為正式節點，原因包括：

```text
可統計完成時間
可統計複雜度
可追蹤整個大型工作的生命週期
可作為子任務完成狀態的彙總點
```

因此，父任務至少需要支援兩種狀態或概念：

```text
拆解完成
實際完成
```

---

### 4. 拆解完成與任務完成需要分開

本次訪談確認：

```text
拆解完成 ≠ 任務完成
```

例如：

```text
設計分散式匯率查詢系統
```

可能已經被拆解成足夠清楚的子任務，但不代表這個系統已經完成。

因此，RonFlow 需要明確區分：

```text
拆解完成：
這個任務已經被拆到足夠清楚，後續可以交由子任務進入執行。

任務完成：
這個任務本身代表的工作成果已經完成，或所有必要子任務已經完成。
```

---

## 六、Ready 判斷與 Flow 進入條件

### 1. Ready 判斷位置

本次訪談確認採用以下方向：

```text
父任務檢查是否拆解完成。
葉節點檢查是否 Ready。
```

也就是：

```text
父任務：重點是是否已經完成拆解。
葉節點：重點是是否已經可以執行。
```

---

### 2. Ready 判斷採輔助提醒，不採強制阻擋

使用者傾向：

```text
輔助提醒
```

RonFlow 可以透過檢核表單提醒任務可能尚未 Ready，但不應過度限制使用者操作。

這表示：

```text
系統提供明確檢核。
系統可顯示任務尚未 Ready。
系統可提示任務仍應留在任務樹中。
但使用者仍保有最終判斷權。
```

---

### 3. 不導入 AI 輔助判斷

本次訪談明確確認：

```text
RonFlow 第一版不導入 AI 輔助判讀。
```

所有判斷應該由明確程式碼邏輯、欄位、表單與使用者操作構成。

因此，不採用以下方向：

```text
AI 自動判斷任務太大
AI 自動拆解任務
AI 自動估算完成時間
AI 自動判斷任務複雜度
```

取而代之的是：

```text
Checklist-based
Rule-based
Explicit-state-based
User-confirmed
```

---

### 4. 任務進入 Flow 前的最低檢核項目

初版 Ready checklist 暫定為：

```text
1. 是否有具體產出物？
2. 是否有明確完成條件？
3. 是否可以明確預估預計完成時間？
```

原本候選項目中，有幾項被使用者認為目前不適合放入最低檢核：

```text
任務的動作是什麼：偏抽象，先拿掉。
第一個執行步驟是什麼：不必要，先拿掉。
是否還有未決策事項：目前暫不納入。
是否還有未釐清依賴：目前暫不納入。
```

其中「是否可以在合理時間內完成」調整為：

```text
是否可以明確預估預計完成時間？
```

此檢核表未來仍可能變動，尚未視為最終定案。

---

### 5. 不 Ready 的任務處理方式

若任務尚未 Ready，RonFlow 應該：

```text
保持在任務樹中，不進 Flow。
```

目前不需要自動轉成調查任務、決策任務或設計任務。

也不需要系統自動強制拆解。

---

## 七、任務分類原則

本次訪談確認，任務類型不宜過多。

不採用複雜分類，例如：

```text
目標
調查
決策
設計
實作
驗收
文件
溝通
```

使用者傾向少一點分類。

第一版可先聚焦在：

```text
尚未 Ready
Ready
已進入 Flow
```

或類似的簡化狀態。

若後續需要，再補充更細的任務性質分類。

## 八、任務樹與視覺呈現

### 1. 底層採任務樹

使用者明確選擇：

```text
樹狀結構：
任務底下有子任務，子任務底下還能有子任務。
```

任務拆解層數：

```text
不限制層數。
```

---

### 2. 大綱、樹狀、Mind Map 本質上可共用同一資料模型

本次討論中確認：

```text
大綱結構
樹狀結構
Mind Map
```

底層都可以是同一種 Node Tree。

差別主要在呈現方式，而不是資料模型。

第一版 MVP 不需要實作 Mind Map。

可先採用：

```text
底層資料模型：Task Tree
MVP 呈現方式：大綱式樹狀結構
未來可能擴充：Mind Map View
```

## 九、MVP 初步範圍

MVP 第一版應聚焦在主要功能：

```text
可不斷拆解子任務，
並經由檢核表單確認該任務已經可執行，
再進入既有 Flow。
```

### MVP 應包含

```text
1. 在專案中建立大型任務。
2. 任務可以建立子任務。
3. 子任務可以繼續建立子任務。
4. 任務樹支援不限層數的遞迴拆解。
5. 任務可以填寫或確認完成條件。
6. 任務可以檢核是否 Ready。
7. 不 Ready 的任務保留在任務樹中。
8. Ready 的任務可以進入既有 Flow。
9. 父任務可以標記為拆解完成。
10. 父任務作為統計節點被保留。
```

---

## 十、MVP 非目標清單

第一版暫不處理以下內容：

```text
AI 自動拆解
AI 自動判斷 Ready
AI 自動估時
AI 自動判斷任務難度
Mind Map 視覺化
複雜報表
甘特圖
跨專案依賴
多人協作
自動複雜度計算
完整統計儀表板
```

其中，報表與統計雖然未來可能重要，但第一版不納入 MVP。

原因是第一版應先確認核心流程是否成立：

```text
建立大型任務
  ↓
遞迴拆解
  ↓
Ready 檢核
  ↓
Ready 任務進入 Flow
```

## 十一、成功判準

若此功能做對，使用者應該可以做到：

```text
1. 可以放心地把大型、抽象、尚未規劃完成的工作放進 RonFlow。
2. 不需要再先用 Markdown 當前置任務整理區。
3. 可以在 RonFlow 中直接拆解大型任務。
4. 子任務可以繼續被拆解，不受層數限制。
5. 任務是否可以執行，可以透過明確檢核表單確認。
6. 尚未 Ready 的任務不會被迫進入 Flow。
7. 已經 Ready 的任務可以回到 RonFlow 既有執行流程。
8. 父任務可以保留為節點，用於後續完成時間、複雜度等統計。
9. 使用者可以用 RonFlow 管理各種大小的任務，而不只是已經整理好的任務。
```

使用者原始成功敘述可整理為：

```text
以前我會琢磨這個任務是否可以直接放到 RonFlow 上，
還是要拆成幾個子任務被分別執行。

現在 RonFlow 可以讓我直接在系統上做事、規劃、分配。

所以我能直接使用系統管理各種大小的任務。
```

## 十二、主驗收案例

本功能的主驗收案例，採用：

```text
RonFlow 需求說明書：抽象任務分解與任務成形管理
```

也就是以 RonFlow Hatchery 功能本身作為驗收案例。

驗收時應確認：

```text
1. 可以建立「抽象任務分解與任務成形管理」作為大型任務。
2. 可以在其下拆出多個子任務。
3. 子任務可以繼續拆解。
4. 父任務可以標記為拆解完成。
5. 葉節點任務可以填寫完成條件。
6. 葉節點任務可以透過檢核表單判斷是否 Ready。
7. 不 Ready 的任務保留在任務樹。
8. Ready 的任務可以送入既有 Flow。
9. 父任務保留統計節點，不因子任務進入 Flow 而消失。
```

## 十三、初步 Domain Glossary

此處只是訪談一的初步詞彙，不代表最終 Domain Model。

### Project

專案。  
RonFlow 中的工作區域或容器。  
專案不是任務，也不是父任務。

### Task Tree

任務樹。  
存在於 Project 底下，用來承載任務與子任務的遞迴拆解結構。

### Task Node

任務節點。  
任務樹中的任一節點，可以是大型任務、子任務或葉節點任務。

### Parent Task

父任務。  
具有子任務的任務節點。  
父任務可以被標記為拆解完成，也可作為統計節點保留。

### Leaf Task

葉節點任務。  
沒有子任務的任務節點。  
通常是 Ready 檢核與進入 Flow 的主要候選對象。

### Decomposition Complete

拆解完成。  
代表父任務已經被拆解到足夠清楚，不等於所有實際工作已完成。

### Ready

可執行。  
代表任務已符合最低檢核條件，可以進入既有 Flow。

### Flow

既有任務執行流程。  
負責 Todo、Active、Review、Done 等執行狀態管理。

## 十四、後續待釐清問題

訪談一已完成問題與使用情境收斂，但以下議題留待後續訪談處理。

### 1. 任務成熟度狀態

需要進一步定義：

```text
尚未 Ready
拆解中
拆解完成
Ready
已進入 Flow
已完成
```

這些狀態是否都需要存在，以及它們之間如何轉換。

---

### 2. 父任務完成規則

目前初步共識是：

```text
父任務可以標記拆解完成。
所有子任務完成後，父任務也可被視為完成。
父任務需要保留節點以利統計。
```

但仍需進一步釐清：

```text
父任務是否可以手動完成？
子任務全部完成時，父任務是否自動完成？
父任務完成時間如何計算？
父任務複雜度如何彙總？
```

---

### 3. Ready Checklist 最終版

目前最低檢核暫定為：

```text
是否有具體產出物？
是否有明確完成條件？
是否可以明確預估預計完成時間？
```

但使用者也指出此題未來可能會變動。

後續需要針對 Ready checklist 進一步討論。

---

### 4. 任務樹與 Flow 的關係

需要進一步釐清：

```text
Ready 任務進入 Flow 後，是否仍保留在任務樹？
Flow 中任務完成後，是否回寫任務樹狀態？
任務樹中的父任務如何計算子任務進度？
```

---

### 5. 統計資料

父任務保留為統計節點，但 MVP 暫不做報表。

後續仍需釐清：

```text
完成時間如何統計？
複雜度如何統計？
父任務統計是否由子任務彙總？
拆解耗時是否要和執行耗時分開？
```

---

## 十五、訪談一結論

本次訪談確認 RonFlow Hatchery 的核心方向是：

```text
RonFlow 應支援大型、抽象、尚未成形的任務直接被建立在系統中，
並透過任務樹進行遞迴拆解，
再透過明確檢核表單判斷葉節點任務是否 Ready，
最後將 Ready 任務送入既有 Flow 執行。
```

第一版不追求視覺化、AI、自動估算、報表或多人協作。

第一版最重要的是驗證核心流程：

```text
大型任務可以被接住
任務可以被遞迴拆解
任務可以被檢核是否 Ready
Ready 任務可以進入 Flow
父任務可以保留為統計節點
```

這代表 RonFlow 的能力將從：

```text
管理已經可以執行的任務
```

擴展為：

```text
管理任務從模糊、拆解、成形，到執行與完成的完整過程
```
